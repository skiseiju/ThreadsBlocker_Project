# ADR-0005 封鎖/檢舉流程中加入機器人特徵偵測

日期：2026-04-20
狀態：已決定

## 背景

ThreadsBlocker 平台頁面需要呈現「AI 機器人」分類資料，但目前封鎖/檢舉流程只記錄帳號名稱，無任何帳號特性資料。mock 資料中的 signals、ai_probability 欄位在真實資料中並不存在。平台頁面的三分類卡片（AI 機器人、政治狂熱者、噁男）缺乏真實數據支撐。

## 考慮過的選項

### 選項 A：伺服器端推斷
上傳後由後端依 reportPath 分類。
- 優點：不改 extension，實作簡單
- 缺點：reportPath 沒有 AI 機器人對應選項，無法識別

### 選項 B：使用者主動標記（UI checkbox）
在封鎖 picker 每個帳號旁加標記按鈕。
- 優點：直接
- 缺點：干擾封鎖主流程，使用者認知負擔高

### 選項 C：Worker 讀 profile + 使用者事後確認（本決定）
封鎖/檢舉 worker 執行過程中順道拜訪被處理帳號的 Threads profile 頁面，抽取可疑特徵，流程結束後彈出確認視窗讓使用者勾選同意。

## 決定

採用選項 C。

理由：
1. Worker 本身已在開 Threads 頁面運作，讀 profile 是搭便車而非額外開銷
2. 事後確認模式不打斷主要流程
3. 使用者確認後的資料是有人類背書的真實標記，比純演算法推斷可信
4. 封鎖與檢舉皆可受益，不限單一流程

## 實作範圍

- Worker 端：拜訪 `threads.net/@{username}`，讀取三無特徵、追蹤比、原創率、發文間隔
- 計算可疑分數（門檻 ≥ 5 才觸發確認視窗）
- 確認 UI：流程結束後彈出，列出達門檻帳號 + 偵測到的特徵，使用者可逐一確認或排除
- Storage：新增 `hege_bot_signals` key，儲存已確認帳號特徵
- Upload：reporter.js 聚合時加入 `confirmedBots` 陣列

## 後果

- 平台頁面三分類卡片取得真實資料基礎
- 需為 worker 增加 profile 讀取時間（每帳號約 1-2 秒，可並行）
- 增加一個使用者互動步驟，需設計為可跳過
- 新增 storage key 需加入 SYNC_KEYS
