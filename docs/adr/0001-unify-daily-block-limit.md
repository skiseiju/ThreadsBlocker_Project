# 0001 統一每日封鎖安全上限

## 背景

原本系統同時存在「延時封鎖（`DELAYED_QUEUE` 100/8h）」與「水庫批次大小（`SWEEP_BATCH_SIZE`）」兩個分散設定。兩者本質上都在處理同一件事：避免 Meta 在短時間內看到太多封鎖動作，降低帳號被限制的風險。

實際使用時，使用者很難理解「延時水庫」與「批次大小」的差別，也不容易判斷該調哪一個設定。這讓封鎖風險控管變成內部實作細節，而不是使用者能直覺掌握的安全選項。

## 選項

- (a) 統一成一個每日上限，並由 worker 自我節流。
- (b) 保留延時水庫與批次大小兩個分散設定。
- (c) 完全移除節流，讓使用者自行承擔風險。

## 決定

採用 (a)：把「延時封鎖（`DELAYED_QUEUE` 100/8h）」與「水庫批次大小（`SWEEP_BATCH_SIZE`）」合併成單一「Meta 每日安全上限（`DAILY_BLOCK_LIMIT`）」加上緊急模式 toggle。

這個設計讓使用者只需要回答一個問題：「我願意一天承擔多少封鎖風險？」其餘節流細節由 worker 依每日上限自行處理。

## 後果

好處是設定語意更單純，風險控管集中在同一個入口，後續也比較容易在 UI 裡清楚呈現。

副作用是原本「先存進延時水庫，再慢慢釋放」的節奏感會消失。使用者需要改從 `BG_QUEUE` 的 ETA 與目前進度觀察系統還要多久才會完成。

未來若要補回更細的節奏控制，應以 `DAILY_BLOCK_LIMIT` 為核心延伸，而不是重新引入另一組平行的批次設定。
