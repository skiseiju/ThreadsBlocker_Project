# Threads DOM 開發避坑指南 (GOTCHAs)

在開發 Threads 網頁版插件（如：「留友封」功能）時，針對其 React 虛擬 DOM 的行為特性，後續開發者應注意以下事項，以避免按鈕重複或範圍偵測失效。

## 1. 「堆疊式」對話框 (Stacked Dialogs)

**現象**：
當使用者從「貼文動態」點進「按讚內容」或「轉發清單」時，Threads **不會**關掉原本的對話框，而是將原本的對話框保留在 DOM 結構的最底层，然後在它上方直接「疊加」一層新的對話框結構。

**避坑指南**：
- **切勿使用全域搜尋**：若使用 `document.querySelector` 搜尋「排序」或「按鈕」，極可能會穿透當前對話框，抓到背景被蓋住的那一層。
- **改用局部隔離 (Scope Isolation)**：永遠從標題標籤 (Header) 往外找最近的「局部邊界」(`closest('[role="dialog"]')` 或 `header.parentElement`)。
- **多標題檢查**：掃描標題時不應在中途 `break`，應持續掃描到最後一個（即 DOM 樹中最深、視覺上最上層的對話框）。

## 2. 「讀取過渡期」的延遲偵測 (Loading States)

**現象**：
點擊深層連結後，對話框會有一段「灰底轉圈圈」的 Loading 狀態。此時 DOM 結構可能已經出現，但標題文字（如「1.2 萬個讚」）或功能元件（如「排序」按鈕）可能還沒填入內容。

**避坑指南**：
- **提高掃描靈靈度**：對於這種動態載入，定時器（`setInterval`）建議設定在 500ms 左右，以降低按鈕長出來的延遲感。
- **偵測寬鬆度**：即使排序按鈕是灰底不能點按，只要它是目標容器，就應該強行補上我們自己的按鈕。

## 3. 防複畫機制 (Anti-Rebuild Logic)

**現象**：
由於 React 可能隨時重新渲染局部組件，直接在 DOM 對象上記錄 `dataset.injected = true` 在跨頁切換時可能會因為「容器共用」而導致誤判。

**避坑指南**：
- **元素存留檢查**：不要只依賴屬性（Attribute），應直接透過 `localCtx.querySelector` 定時檢查我們親手畫上去的人造元件（如 `.hege-block-all-btn`）是否還「實體存在」於目前的局部空間內。
- **ID 繫結 (可選)**：針對需要精準同步的元件，可對 Header 和 Button 進行唯一的 ID 靈魂綁定。

## 4. 檢舉流程也有平台上限，且 UI 可能直接封鎖入口

**現象**：
Threads 的「檢舉」不只可能有每日或短時段平台上限；實測中，當帳號進入某種限制狀態時，流程不一定會走到明確的 rate-limit 文案，而是可能直接出現在 chooser 或 menu 階段「看得到檢舉帳號，但點不下去／根本不讓你點」。

**避坑指南**：
- **不要先假設是 DOM 命中錯誤**：遇到「檢舉帳號」卡住時，要先把「平台限制直接封入口」列為同等可能原因。
- **記錄 click 前後狀態**：至少保留 target 元素、可見選項、dialog 數量，以及 click 後 chooser 是否消失。
- **區分兩種失敗**：`找不到檢舉帳號` 跟 `找到檢舉帳號但平台不讓點` 不是同一類 bug。
- **不要把這類 case 誤判成純 selector 問題**：若多次命中同一 target 且 UI 毫無狀態變化，應優先懷疑 Threads 限流／風控，而不是無限疊加 click workaround。
